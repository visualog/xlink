구현 계획

HTTP 우선 구조로 작게 시작하고, 이후 MCP adapter로 확장합니다.

Phase 1
로컬 coordinator 서버 구축

`server.js`, `store.js`, `routes` 구조로 시작합니다. JSON store를 사용해 handoff 생성, 조회, claim, complete를 구현합니다.

Phase 2
artifact와 메시지 기록 확장

썸네일, 코드 스니펫, Figma export 경로를 handoff에 붙이고, 진행 메모와 blocked 이유를 append 할 수 있게 합니다.

Phase 3
MCP adapter 추가

기존 HTTP API를 감싸는 도구 레이어를 추가해 여러 에이전트가 같은 인터페이스로 handoff를 다루게 합니다.

Phase 4
검색, 우선순위, 채널 확장

`priority`, `channel`, `status` 필터를 강화하고 `figma`, `docs`, `review` 같은 채널을 넓혀갑니다.

POST /handoffs 새 handoff 생성과 기본 메타데이터 저장
GET /handoffs 채널, 상태, 우선순위 조건으로 목록 조회
POST /handoffs/:id/claim 담당 에이전트가 작업을 점유하고 `claimedAt` 기록
POST /handoffs/:id/complete 완료 메모를 남기고 `completedAt` 갱신
POST /handoffs/:id/artifacts 경로 기반 artifact 첨부와 참고 자료 연결

핵심 규칙

이 페이지에서 꼭 기억할 포인트만 남겼습니다.

pending claimed completed blocked rejected
JSON first 초기에는 사람이 읽고 디버깅하기 쉬운 저장 방식으로 시작
One claim 한 handoff는 한 에이전트가 claim 하는 원칙 유지
Artifact paths 썸네일, 코드, Figma 결과물을 상대 경로로 첨부
MCP later 먼저 HTTP API를 만들고 그 위에 MCP adapter를 추가
1차 완료 기준은 단순합니다. bridge가 handoff를 만들고, devlog가 pending을 읽고 claim 한 뒤, 기록을 남기고 complete 처리하면 됩니다.
